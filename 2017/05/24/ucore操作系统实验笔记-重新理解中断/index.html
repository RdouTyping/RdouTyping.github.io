ucore操作系统实验笔记 - 重新理解中断

发表于 2017-05-24 | 本文总阅读量 次

(03/08/2017更新) 在上一篇文章ucore操作系统实验笔记 - Lab1中，我已经比较详细地记录了中断的使用。那篇文章关于中断的重点是如何使用IDT、中断描述符和中断向量表等。这篇文章我将把重点放到另外一个地方，也就是中断的过程中如何保存和恢复现场。 现在实验已经做到Lab5了，我对中断有了一些新的的理解，因此我准备重新梳理下中断相关知识。

中断的类型

在x86系统中，中断一般分为3种：

  1. 和CPU无关的，比如外设的请求等，这些属于Interrupt，这种中断是非同步的。
  2. 和CPU有关的，比如除0，page fault等，这些属于Exception，这种中断是同步的。
  3. 系统调用，这些属于Trap。

Interrupt和Exception我们都可以认为它们是Hard Interrupt，Trap被称为Soft Interrupt，后面我会讲它们的区别。

IDT

IDT是Interrupt Descriptor Table的简写，我们使用中断向量号访问IDT从而找到中断号对应的中断服务例程。IDT和GDT类似，由一系列的中断门描述符(Interrupt Gate Descriptor)组成，每个门描述符都是8K。这些中断门描述符可以分为3类：

  1. Task Gate
  2. Interrupt Gate
  3. Trap Gate

【补充】所谓“自动禁止”，指的是CPU跳转到interrupt gate里的地址时，在将EFLAGS保存到栈上之后，清除EFLAGS里的IF位，以避免重复触发中断。在中断处理例程里，操作系统可以将EFLAGS里的IF设上,从而允许嵌套中断。但是必须在此之前做好处理嵌套中断的必要准备，如保存必要的寄存器等。二在ucore中访问Trap Gate的目的是为了实现系统调用。用户进程在正常执行中是不能禁止中断的，而当它发出系统调用后，将通过Trap Gate完成了从用户态（ring 3）的用户进程进了核心态（ring 0）的OS kernel。如果在到达OS kernel后禁止EFLAGS里的IF位，第一没意义（因为不会出现嵌套系统调用的情况），第二还会导致某些中断得不到及时响应，所以调用Trap Gate时，CPU则不会去禁止中断。总之，interrupt gate和trap gate之间没有优先级之分，仅仅是CPU在处理中断时有不同的方法，供操作系统在实现时根据需要进行选择。

简单来说就是Interrupt发生时会自动禁止中断(清除EFLAGS中的IF位)，而Trap发生时不会这样。

当我们建立好IDT后，我们就需要告诉CPU我们将IDT存在了哪里。在ring 0 特权级，也就是处于kernel态时，我们可以通过lidt将IDT的地址和大小加载到IDTR寄存器中，之后CPU就能通过访问该寄存器来获取IDT的起始地址了。下面是IDTR寄存器的示意图：

中断的访问流程

我们先来看看指导书是怎么说中断的执行流程(开始部分)：

  1. CPU在执行完当前程序的每一条指令后，都会去确认在执行刚才的指令过程中中断控制器（如：8259A）是否发送中断请求过来，如果有那么CPU就会在相应的时钟脉冲到来时从总线上读取中断请求对应的中断向量；
  2. CPU根据得到的中断向量（以此为索引）到IDT中找到该向量对应的中断描述符，中断描述符里保存着中断服务例程的段选择子；
  3. CPU使用IDT查到的中断服务例程的段选择子从GDT中取得相应的段描述符，段描述符里保存了中断服务例程的段基址和属性信息，此时CPU就得到了中断服务例程的起始地址，并跳转到该地址；
  4. CPU会根据CPL和中断服务例程的段描述符的DPL信息确认是否发生了特权级的转换。比如当前程序正运行在用户态，而中断程序是运行在内核态的，则意味着发生了特权级的转换，这时CPU会从当前程序的TSS信息（该信息在内存中的起始地址存在TR寄存器中）里取得该程序的内核栈地址，即包括内核态的ss和esp的值，并立即将系统当前使用的栈切换成新的内核栈。这个栈就是即将运行的中断服务程序要使用的栈。紧接着就将当前程序使用的用户态的ss和esp压到新的内核栈中保存起来；
  5. CPU需要开始保存当前被打断的程序的现场（即一些寄存器的值），以便于将来恢复被打断的程序继续执行。这需要利用内核栈来保存相关现场信息，即依次压入当前被打断程序使用的eflags，cs，eip，errorCode（如果是有错误码的异常）信息；
  6. CPU利用中断服务例程的段描述符将其第一条指令的地址加载到cs和eip寄存器中，开始执行中断服务例程。这意味着先前的程序被暂停执行，中断服务程序正式开始工作。

下面是中断的执行流程(结束部分)：

  1. 程序执行这条iret指令时，首先会从内核栈里弹出先前保存的被打断的程序的现场信息，即eflags，cs，eip重新开始执行；
  2. 如果存在特权级转换（从内核态转换到用户态），则还需要从内核栈中弹出用户态栈的ss和esp，这样也意味着栈也被切换回原先使用的用户态的栈了；
  3. 如果此次处理的是带有错误码（errorCode）的异常，CPU在恢复先前程序的现场时，并不会弹出errorCode。这一步需要通过软件完成，即要求相关的中断服务例程在调用iret返回之前添加出栈代码主动弹出errorCode。

有了这些信息，我们基本就对中断有了一些定性的了解了。接下来我们仔细看看每一个步骤具体做了什么。

产生中断向量

这一步是硬件产生的，CPU会不断的检查是否有中断产生。如果发现有新的中断产生了，它就会获取该中断向量。并用该中断向量去IDT中寻找相应的中断门描述符。

寻找中断门描述符

CPU获取中断向量后，它就可以用这个中断向量作为IDT的索引去寻找对应的中断门描述符了。

特权级的检测

在CPU获取到中断门描述符后，首先要做的就是检测特权级，那该怎么检测呢？我们先来看一幅图：

首先，我们需要了解几个名词 - CPL和DPL：
CPL是current privilege level，它的值一般等于当前段描述符的DPL，它存储在%cs中的bit1 : bit0；DPL是descriptor privilege level，注意这里有两个DPL，分别是Gate的DPL和目标段描述符的DPL。
这里还有RPL(requestor’s privilege level)，它存储在%ds, %es, %ss等中，和CPL一样，也是bit1 : bit0(所有的选择子中都有RPL，只是我们习惯叫%cs中的RPL为CPL)。

一般情况下CPL, RPL和DPL要满足以下公式：

MAX(CPL, RPL) <= 段描述符DPL

加上RPL是为了多一层保护，比如CPL为0而%ds中的RPL为3，那么要想访问DPL为0的段选择子(%cs)就不行。如果CPL为3而%ds中的RPL为0，同样不能访问DPL为0的段选择子(%cs)。

除了检测CPL, RPL和DPL外，这里还要做一些特权级的检测，比如CPL和目标段描述符DPL的比较，CPL和Gate DPL的比较等，下面看看具体的规则：

  1. CPL必须要大于等于目标段描述符的DPL，也就是说，中断只能朝着更高特权级或者平级的方向移动。
  2. 如果是INT x造成的中断(软中断)，那么CPL必须小于等于Gate的DPL。利用这点，一般我们会将除了系统调用外的所有中断门的DPL都设置为0，将系统调用的中断门的DPL设置为3。这样用户态使用INT x时，只有使用了正确的x才能成功完成系统调用。
  3. 如果是硬中断的话，不用判断CPL和Gate的DPL，用户进程能直接通过中断进入内核态。

注意，现在的操作系统并没有怎么使用段的特权级来限制用户态访问内核空间。不管是用户态还是内核态，不管是数据段还是代码段，所有的段都是用相同的起始地址和大小，因此这也被称为”flat memory”。也就是说即使在用户态，只要正确地设置CPL, RPL和DPL，CPU依然可以访问所有的内存空间。现在的操作系统都是通过分页机制实现特权级的限制。在中断中，其实特权级检测也没太大作用，只要正确的设置就不会出错，要出错除非将IDT设置错了，或者%cs和%ds设置错了(比如%cs是用户态而%ds是内核态)。

CPU压栈

如果特权级的检测没有问题，那么CPU就会将一些寄存器压入进程的堆栈中，CPU的压栈情况如下图所示：

这是发生中断并且特权级发生转换后栈空间变化的示意图，在压入寄存器前，CPU需要更新%ss和%esp(需要使用TSS)，使其指向新的栈空间。对于不发生特权级转换的中断，有二个地方不同，第一，它只用到一个栈，也就是说Procedure和Handler用的是同一个栈，CPU不需要更新%ss和%esp；第二，CPU不需要压入SS和ESP。除此之外，这两种情况都需要压入CS，EIP和Error Code(如果有的话)。

寻找中断处理例程的入口

CPU通过中断向量号找到IDT中中断门描述符，中断门描述符中存有目标段选择子以及中断处理例程的offset。我们将目标段选择子以及中断处理例程的offset分别载入%cs和%eip，从而使CPU跳到中断处理例程的入口，同时设置CPL == 目标段描述符的DPL。下面是一个流程示意图：

保存中断前的各种寄存器

CPU负责将%cs, %eip, %ss, %esp EFLAGS等信息压入堆栈，而其他的一些信息，比如错误码、中断向量、通用寄存器(pushal)、%ds, %es等就需要内核来完成。

执行中断服务例程

根据中断向量的信息，执行相应的中断服务例程。

当中断服务例程结束后，我们就需要进行中断的返回了。

弹出栈中存储的寄存器等信息

内核将错误码、中断向量、通用寄存器(popal)、%ds, %es, %ss等弹出，然后通过iret来将%cs, %eip, %ss, %esp EFLAGS等弹出。这样，我们就恢复了中断前的状态。
